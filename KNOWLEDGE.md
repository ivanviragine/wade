## 2b4d5319 | 2026-04-11 | implementation | Issue #272

When knowledge commands run from a linked git worktree, load_config() finds
the .wade.yml copied to the worktree and sets project_root to the worktree
directory. The fix: detect the main checkout using git worktree list
--porcelain and redirect all knowledge reads/writes there.
_canonical_project_root() handles expected operational errors (ImportError,
GitError, OSError) and falls back to project_root; unexpected exceptions
propagate so real bugs are not silently swallowed. Apply at the top of service
functions (append_knowledge, get_annotated_knowledge) and in CLI commands
that resolve paths directly (resolve_canonical_knowledge_path wrapper).

## cae0f47a | 2026-04-24 | plan | tags: hooks, claude-code

Claude Code's PreToolUse hook JSON schema only allows 'hookSpecificOutput' at the root level. Extra root-level fields ('permission', 'permissionDecision', 'decision', 'reason') cause strict schema validation failure: 'Hook JSON output validation failed — (root): Invalid input', which makes the hook fail open (write is allowed). The correct deny output is: {"hookSpecificOutput": {"hookEventName": "PreToolUse", "permissionDecision": "deny", "permissionDecisionReason": "..."}}

## cedeaad0 | 2026-04-24 | plan | tags: plan-session, implementation-session, skills, review

Review cycles should be capped at 2 total runs: 1 run if findings are minor (fix and proceed without re-review); 2 runs if findings are major (fix and re-run once, then always proceed). This prevents agents from looping indefinitely. The cap is enforced via text in _partials/review-plan-step.md and _partials/review-implementation-closing-step.md.

## ddc10718 | 2026-04-27 | implementation | tags: plan-session, prompts, ai-behavior | Issue #281

When writing instructions for AI tools, "ask the user X" is not sufficient to prevent the tool from using native UI components (e.g. AskUserQuestion with pre-defined options). You must explicitly say "output a plain text question" and "do NOT use a native selection/question component or present pre-defined categories."

## 6264c1a6 | 2026-04-27 | plan | tags: github, gh-cli, labels

`gh label list --search <name>` is unreliable for label-existence checks — it returns empty for label names that exist (verified for `bug`, `complexity:medium`, `review-addressed-by:claude`). Use `gh api repos/{slug}/labels/{url-encoded-name}` with `check=False` instead; it returns 200 if the label exists and 404 if not.
